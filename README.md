# AIPM Skills

AIPM Skills 是一套面向 AI 产品项目设计的 Codex Skill 工作流。

它的目标不是教用户泛泛地“做一个 AI 项目”，而是把一个 AI 项目从想法、业务场景、功能链路、实现方案、评测体系到 bad case 迭代，拆成可以被 Agent 一步步带着完成的过程。

这套流程适合用于：

- AI 产品经理作品集项目
- AI 产品实习 / 面试项目包装
- RAG、Agent、AI workflow、AI 工具类项目的场景重构
- 从 Vibe Coding demo 升级成更像真实业务功能的 AI 产品
- 需要解释“为什么这个项目需要 AI、怎么实现、怎么评测”的项目复盘

## 整体闭环

这套 Skill 目前包含三个阶段：

```text
aipm-anchor
-> aipm-chain
-> aipm-eval
```

三步合起来回答一个完整 AI 项目的核心问题：

```text
1. 这个项目到底锚定哪个真实业务场景？
2. AI 应该介入哪条功能链路？
3. 这条链路每一步怎么实现、怎么沉淀状态、怎么形成闭环？
4. 怎么评测它是否有效？
5. 出现 bad case 后，怎么归因、修改、复测？
```

换句话说，这不是一组孤立 prompt，而是一条 AI 产品项目的完整证据链：

```text
业务场景
-> 高摩擦任务节点
-> AI 介入点
-> MVP 功能定义
-> 主链路 L1-L8
-> 节点级实现方案
-> 离线评测体系
-> bad case 归因
-> 同口径复测
```

## 01. AIPM-Anchor

路径：

```text
skills/aipm-anchor
```

`aipm-anchor` 用来完成 AI 项目的第一步：定义项目到底应该做哪个 AI 功能。

它不从页面、功能列表、模型能力或技术栈开始，而是先判断：

- 有没有真实业务场景
- 用户在什么触发时刻需要这个功能
- 哪个任务节点足够高摩擦
- 这个节点为什么需要 AI
- 是否能沉淀反馈、bad case 或状态
- 是否存在主动服务的触发信号
- 第一版 MVP 能不能在 3-7 天内跑出证据

核心产出：

- 现有功能诊断
- 业务场景与 AI 节点锚定
- MVP 功能定义
- 自进化接口
- 主动服务接口
- 关键约束

适合在这些情况下使用：

- 用户只有一个模糊 AI 项目想法
- 项目像一个 demo，但不像真实业务功能
- 简历项目里写了 AI，但说不清 AI 必要性
- 项目功能很多，但没有主场景
- 想把工具型项目改造成更有产品深度的 AI 项目

## 02. AIPM-Chain

路径：

```text
skills/aipm-chain
```

`aipm-chain` 用来完成 AI 项目的第二步：把已经锚定的 AI 功能拆成可实现、可解释、可迭代的主链路。

它不直接跳到 RAG、Agent、数据库、前后端或模型选择，而是先把功能拆成一条从触发到下一次服务的链路：

```text
L1 触发
-> L2 输入
-> L3 预处理
-> L4 核心处理
-> L5 输出
-> L6 反馈
-> L7 状态沉淀
-> L8 下一次/主动触发
```

然后逐层下钻每个环节：

- 输入是什么
- 处理动作是什么
- 输出给谁
- 哪些状态要保存
- 哪些地方会失败
- 第一版怎么用最小方式实现
- 后续什么情况下再升级技术复杂度

核心产出：

- 主功能链路地图
- L1-L8 链路说明
- 每个节点的输入、处理、输出、边界、状态沉淀
- MVP 实现取舍
- 3-7 天最小证据闭环

适合在这些情况下使用：

- 用户已经知道要做哪个 AI 功能，但不知道怎么拆实现
- 项目描述停留在“用 AI 生成/分析/推荐”
- 需要把 AI 功能写成面试时能解释清楚的技术链路
- 需要判断哪些部分用 AI，哪些部分用规则、表格、人工审核或简单存储即可
- 需要避免一上来就堆复杂 Agent 架构

## 03. AIPM-Eval

路径：

```text
skills/aipm-eval
```

`aipm-eval` 用来完成 AI 项目的第三步：基于主链路搭建一套能验证效果、定位问题、驱动迭代的评测体系。

它不做通用模型评测科普，也不追求虚假的指标漂亮，而是围绕一条具体 AI 功能链路回答：

```text
这个 AI 功能怎样才算好？
如果坏了，应该改哪一环？
改完之后，怎么用同一套口径复测？
```

它会带用户完成：

- 继承第二阶段主链路
- 明确评测目的：验收、迭代、边界压测
- 选择评测对象：链路节点、端到端用户结果，或两者都评
- 构建 eval case 表
- 设计 MECE rubric
- 区分人工评分和 LLM-as-a-judge
- 生成 bad case 归因示例
- 定义复测口径
- 明确项目表达边界

特别注意：`aipm-eval` 不会替用户编造真实评测结果。

如果用户还没有实际跑评测，它只会输出：

- 可能出现的 bad case 示例
- 可能的低分维度
- 可能的链路归因
- 示例修改动作
- 复测方式示范

并且会明确说明：

```text
以下是可能出现的 bad case 示例和归因写法示范，不是已经发生的真实评测结果。
真实报告里只能保留实际跑样本后出现的 case、分数和修改记录。
```

核心产出：

- 评测对象定义卡
- 评测集设计表
- Eval case 数据表
- MECE 评分标准
- 人机协同评测流程
- Bad case 记录表
- 复测方案
- 项目表达边界

适合在这些情况下使用：

- 用户已经有 AI 功能链路，但不知道怎么证明它有效
- 项目需要从“我觉得效果不错”升级成“我有样本、rubric、bad case 和复测”
- 简历或面试里需要解释项目如何评测和迭代
- 需要把低分结果归因到 prompt、检索、schema、规则、反馈或状态沉淀
- 需要避免把离线评测夸成线上效果

## 推荐使用顺序

### 从零开始做 AI 项目

```text
1. 用 aipm-anchor 找到真实业务场景和 AI 介入节点
2. 用 aipm-chain 拆出主链路和实现方案
3. 用 aipm-eval 搭建评测体系和 bad case 迭代方式
```

### 已经有项目 demo

```text
1. 先用 aipm-anchor 诊断这个 demo 是否只是功能堆叠
2. 如果场景成立，再用 aipm-chain 梳理主链路
3. 最后用 aipm-eval 生成评测体系，补上项目可信度
```

### 已经有功能链路

```text
1. 可以跳过 aipm-anchor
2. 直接用 aipm-chain 细化 L1-L8
3. 再用 aipm-eval 做离线评测、rubric 和复测方案
```

### 已经有实现，但没有评测

```text
1. 用 aipm-chain 反向梳理当前系统链路
2. 用 aipm-eval 选择关键节点和端到端结果
3. 构建 20-50 条 eval case，跑第一版 bad case 归因
```

## 最终项目材料应该能回答的问题

完成三步后，一个 AI 项目至少应该能说清楚：

- 目标用户是谁
- 用户在什么场景下遇到什么高摩擦任务
- 为什么这个任务节点需要 AI
- 第一版 MVP 解决哪条主链路
- 主链路每一步输入、处理、输出是什么
- 哪些地方由 AI 处理，哪些地方由规则、人工或简单存储处理
- 系统如何收集反馈和 bad case
- 哪些状态会影响下一次服务
- 用什么样本和 rubric 评测
- 低分样本怎么归因到链路节点
- 修改后怎么复测
- 哪些指标可以写，哪些暂时不能写

## 项目表达边界

这套 Skill 的一个重要原则是：不夸大证据。

可以写：

- 构建离线评测集
- 设计 MECE rubric
- 使用人工标注和 LLM-as-a-judge 预评
- 基于 bad case 归因优化链路
- 使用同一套样本和评分标准复测

谨慎写：

- 准确率提升
- 满意度提升
- 采纳率提升
- 任务完成率提升

这些说法需要样本量、baseline、版本对照、评分口径和人机评分方式。

暂时不能写：

- 已证明真实线上效果
- 转化率或留存提升
- 服务真实用户数量
- 自动学习或自动优化
- 可规模化上线

除非项目确实有真实日志、真实用户、统计周期和可解释的归因方式。

## Repository Structure

```text
skills/
  aipm-anchor/
    SKILL.md
    agents/openai.yaml
  aipm-chain/
    SKILL.md
    agents/openai.yaml
  aipm-eval/
    SKILL.md
    agents/openai.yaml
```

每个 Skill 目录包含：

- `SKILL.md`：Codex 实际读取的技能说明和工作流
- `agents/openai.yaml`：Codex UI 中展示的名称、简介和默认 prompt

## Design Principle

AIPM Skills 的核心不是让 Agent 输出更多内容，而是让 Agent 按正确顺序追问、判断和收敛。

这套流程始终优先保证：

- 业务场景真实
- AI 介入点必要
- 功能链路可解释
- MVP 证据可收集
- 评测标准可复用
- bad case 可归因
- 项目表达不夸大

最终目标是让一个 AI 项目不仅“看起来用了 AI”，而是能被清楚地解释：

```text
为什么做
做什么
怎么做
怎么评
坏了怎么改
改完怎么证明变好
```
